# Dotisan Resource Management Enhancement PRD

## Status: Draft

## 1. Overview

### Goals
- Fix critical logic bug where unmanaged files are incorrectly marked "in sync"
- Add proper drift detection for Homebrew packages (detect external changes)
- Support indexed import syntax for managing individual items in lists
- Unify all resource types to use consistent list-based structure
- Improve import UX with Terraform-style warnings and code snippets

### Background
Dotisan currently has inconsistent behavior:
1. Files without state but with existing destination are marked "in sync" (should be "+ to create")
2. Homebrew drift not detected - can't tell if packages were uninstalled externally
3. Import syntax doesn't support lists (can't import `BrewPackages[ripgrep]`)
4. Resource models are inconsistent (files = single, packages = aggregate)

## 2. Functional Requirements

### FR1: Fix Core Logic Bug
**User Story:** As a user, when a destination file exists but the resource is not in state, I should see it as "+ to create" with a warning, not hidden as "in sync".

**Current Behavior:**
```
Plan: +1 to add  (only shows test-template-file)
```

**Expected Behavior:**
```
Plan: +2 to add

Resources to be created
  ✚ ManagedFile/zshrc will be created
    ⚠ Warning: Destination file already exists at ~/.zshrc
    ℹ To adopt the existing file: dotisan state import ManagedFile zshrc ~/.zshrc
  
  ✚ ManagedFile/test-template-inline will be created
```

### FR2: Homebrew Drift Detection
**User Story:** As a user, I want to know if Homebrew packages were uninstalled outside of dotisan so I can restore them.

**Current:** No drift detection for brew packages
**Expected:** 
```
Drifted resources (manual changes detected)
  ⚠ BrewPackages/core-tools has drift
    Package 'ripgrep' was uninstalled outside of dotisan
    Run 'dotisan apply' to restore
```

**Implementation:** Use `brew info --json ripgrep htop` to check only expected packages (faster than all installed).

### FR3: Indexed Import Syntax
**User Story:** As a user managing lists of packages, I want to import individual items using index syntax.

**Current:**
```bash
dotisan state import ManagedFile zshrc ~/.zshrc
```

**Expected:**
```bash
# Import specific item from a list resource
dotisan state import BrewPackages core-tools[ripgrep] ripgrep
dotisan state import BrewPackages core-tools[0] ripgrep  # index-based
```

### FR4: Unified List Structure
**User Story:** As a developer, I want all resource types to use consistent list-based structures.

**Current Inconsistency:**
- `ManagedFile` = single file
- `ManagedDirectory` = single directory  
- `BrewPackages` = list of formulae/casks
- `NpmPackages` = list of packages

**Expected Consistency:**
All resource types support lists:
```yaml
# ManagedFile can define multiple files
apiVersion: github.com/wasilak/dotisan/v1
kind: ManagedFile
metadata:
  name: dotfiles
spec:
  files:
    - sourceFile: shell/.zshrc
      destination: ~/.zshrc
    - sourceFile: shell/.bashrc
      destination: ~/.bashrc
```

## 10. Duplicate Commands Issue

### Problem
Two commands perform identical operations:
- `dotisan eject KIND NAME` - "Stop managing a resource"
- `dotisan state remove KIND NAME` - "Remove resource from state only"

Both:
- Remove the resource from state file
- Leave the actual resource untouched on the system
- Require KIND and NAME arguments

### Current Differences
- `eject` has `--force` flag to skip confirmation
- `eject` has more detailed output and confirmation prompt
- `state remove` is under `state` subcommand (more discoverable)

### Proposed Resolution
**Option A: Consolidate under `state remove`**
- Remove `eject` command entirely
- Add `--force` flag to `state remove`
- Update docs to use `state remove` exclusively

**Option B: Differentiate functionality**
- `eject` = Remove from state + optionally delete from system (destructive)
- `state remove` = Remove from state only (current behavior for both)

**Recommendation: Option A** - Keep it simple, single command for single purpose

### Implementation
- [ ] Mark `eject` as deprecated with warning
- [ ] Add `--force` flag to `state remove`
- [ ] Update documentation
- [ ] Remove `eject` in v2.0
